模拟信号与 ADC：温度是连续的，芯片只能读数字

时长 ≈ 2h · 中级阶段
🌡️
心智图像：真实世界是连续的水流（温度、光线、声音都是慢慢变的）；而芯片是个「只会数整数」的人。ADC（模数转换）就是个翻译官：它每隔一小会儿看一眼水流大小，记成一个整数（ESP32 是 0~4095）。采得越密、刻度越细，翻译越准。

GPIO 数字脚只能读 0/1，但 ESP32 有一批带 ADC 的脚（标了 ADC1/ADC2），能把 0~3.3V 的模拟电压翻译成 12 位数字（0~4095）。下面用旋钮（电位器）做例子：拧动它，电压连续变化，ADC 把它变成数字。

from machine import ADC, Pin
adc = ADC(Pin(34))          # GPIO34 接电位器中脚
adc.atten(ADC.ATTN_11DB)        # 量程扩到 0~3.3V
adc.width(ADC.WIDTH_12BIT)       # 12 位: 0~4095

while True:
    raw = adc.read()              # 0~4095
    volt = raw / 4095 * 3.3   # 换算成电压
    print("原始:", raw, "电压:", round(volt,2), "V")
    time.sleep(0.3)
交互 · ADC 采样与量化拖滑块改采样率
🛠️排错卡 · ADC 读数异常
症状

读数一直卡在 4095 或 0，或者数值疯狂乱跳。

原因

① 电位器中脚没接到 ADC 脚；② 没设 atten(11DB)，量程只有 0~1V；③ 电源噪声 / 接触不良。

解法

确认中脚接 GPIO34；adc.atten(ADC.ATTN_11DB) 扩到 0~3.3V；软件上多次采样取平均；电源端加个 0.1µF 去耦电容。

接线的三根线电位器：左脚接 3V3，右脚接 GND，中脚（滑臂）接 ADC 脚（如 GPIO34）。拧动时中脚在两个极端之间分压。

PWM：用「快速开关」模拟调光，做呼吸灯

时长 ≈ 2h · 中级阶段
🔆
心智图像：LED 只能「全亮」或「全灭」，没法直接调亮度。但如果你一秒里开关几千次，亮的时间占 70%、灭的占 30%，人眼就把平均看成「七成亮」。这个「亮的时间占比」叫占空比（Duty Cycle）。PWM 就是自动帮你高速开关的那个机构。

PWM（脉冲宽度调制）几乎是控制亮度、电机转速、舵机角度的万能钥匙。MicroPython 里用 machine.PWM，设置频率（每秒开关多少次）和占空比（0=全灭，最大值=全亮）。

from machine import PWM, Pin
import time

pwm = PWM(Pin(2))      # 在 GPIO2 上产生 PWM
pwm.freq(1000)            # 每秒开关 1000 次

while True:
    for d in range(0, 65536, 800):   # 占空比由 0 升到满
        pwm.duty_u16(d)            # 16 位: 0~65535
        time.sleep_ms(5)
    for d in range(65535, 0, -800):  # 再降回去
        pwm.duty_u16(d)
        time.sleep_ms(5)
交互 · PWM 呼吸灯拖动改变占空比
🛠️排错卡 · PWM 不渐变 / 频闪
症状

LED 不会平滑呼吸，而是突然全亮/全灭，或肉眼可见地闪。

原因

① PWM 频率太低（如 1Hz）肉眼能看到闪烁；② 占空比 API 版本错（duty vs duty_u16）；③ 该脚不支持 PWM。

解法

频率设 1000Hz 以上；按 MicroPython 版本用对 API；确认脚支持 PWM（GPIO2/4/5/12~19/21~23 等）；先 print(pwm) 看报错对号入座。

占空比的「最大值」因版本而异老版 MicroPython 用 pwm.duty(0~1023)；新版用 pwm.duty_u16(0~65535)。你先 print(pwm) 看报错提示即可对上号。

连接 Wi‑Fi：让芯片「上网」

时长 ≈ 2h · 中级阶段
📶
心智图像：连 Wi‑Fi 就像 joining 一个会议室：先报上房间名（SSID），再输密码（PASSWORD），路由器同意了就给你发个「工牌」——也就是 IP 地址。有了 IP，你才能跟外面的世界（手机、服务器、互联网）对话。

ESP32 的无线是硬件自带，软件里用 network 模块。注意：连接是异步的，要「等」它连上再往下走，不然马上用网络会报错。

import network, time
wlan = network.WLAN(network.STA_IF)   # STA=作为设备连路由器
wlan.active(True)
wlan.connect("你的WiFi名", "密码")

while not wlan.isconnected():
    print("连接中...")
    time.sleep(1)

print("已连接! IP =", wlan.ifconfig()[0])
交互 · Wi‑Fi 连接状态机点「连接 / 断网」
家庭 Wi‑Fi 大多是 2.4GHzESP32 只支持 2.4GHz，连不上 5GHz 的 Wi‑Fi 是正常现象。另外把密码写死在代码里只是练习，真实产品要用后面「配网」技巧。
🛠️排错卡 · Wi‑Fi 连不上
症状

一直打印「连接中…」，isconnected() 永远 False。

原因

① 路由器是 5GHz（ESP32 只认 2.4G）；② SSID/密码写错；③ 没先 active(True) 再 connect；④ 信号太弱或隐藏 SSID。

解法

确认连 2.4G 网络；核对账号密码；顺序是 active→connect→等 isconnected 再发网络请求；把板子靠近路由器；隐藏 SSID 需显式传参数。

做个 Web 服务器：用手机控制 LED

时长 ≈ 2.5h · 中级阶段
📱
心智图像：ESP32 自己变成了一个迷你网站。你的手机浏览器打开它的 IP，就像访问任意网页。你点页面上的「开/关」，手机就发一条 HTTP 请求「/on」或「/off」，芯片收到后翻转 LED——完全不需要装 App。

用 MicroPython 的 socket 就能搭一个极简 HTTP 服务器。它监听 80 端口，收到请求后看路径决定动作，再回一段 HTML（含两个按钮）。

import socket, network
from machine import Pin
led = Pin(2, Pin.OUT)

def page():
    s = '<a href="/on">开灯</a> <a href="/off">关灯</a>'
    return "HTTP/1.1 200 OK\r\nContent-Type: text/html\r\n\r\n<h2>ESP32</h2>"+s

s = socket.socket()
s.bind(('0.0.0.0', 80))
s.listen(1)
while True:
    c, _ = s.accept()
    req = c.recv(1024).decode()
    if "/on" in req: led.value(1)
    if "/off" in req: led.value(0)
    c.send(page().encode())
    c.close()
交互 · Web 服务器请求流点开灯/关灯看数据流动
连不上？手机要和 ESP32 在同一个 Wi‑Fi 下；浏览器地址填 http://<ESP32的IP>。若提示「拒绝连接」，多半是服务器死循环没 accept 到，检查代码缩进。
🛠️排错卡 · 手机打不开网页
症状

浏览器打不开 http://ESP32的IP，提示拒绝连接或超时。

原因

① 手机和 ESP32 不在同一 Wi‑Fi；② IP 地址填错；③ 服务器死循环没 accept 到新连接；④ 80 端口被占 / 防火墙。

解法

确认同网段；地址用 http://<打印出的IP>；检查 accept() 缩进在 while 里；换端口（如 8080）；关掉电脑防火墙试。

I2C 与 OLED 显示屏：让数据有「脸」

时长 ≈ 2h · 中级阶段
🎓
心智图像：I2C 像课堂上老师点名。整条总线只有两根线（时钟 SCL + 数据 SDA），可以挂很多学生(设备)。老师(ESP32 主机)先喊「3C 号同学」(地址)，只有被点到的屏才应答，然后双方用这两根线交换数据。其它屏假装没听见。

最常见的 0.96 寸 OLED 屏地址是 0x3C。用 machine.I2C 建立总线（默认 SCL=GPIO22, SDA=GPIO21），再用 SSD1306 驱动库画图。下面先「扫描」确认地址。

from machine import I2C, Pin
from ssd1306 import SSD1306_I2C

i2c = I2C(0, scl=Pin(22), sda=Pin(21))
print("发现的设备:", [hex(x) for x in i2c.scan()])  # 通常打印 [0x3c]

oled = SSD1306_I2C(128, 64, i2c)
oled.text("Hello ESP32", 10, 20)
oled.show()
交互 · I2C 总线寻址看主机如何点名
🛠️排错卡 · OLED 不亮 / I2C 找不到
症状

屏不亮，或 i2c.scan() 返回空列表 []。

原因

① I2C 地址不对（常见 0x3C，也有 0x3D）；② SDA/SCL 接反；③ 用了 5V 供电（屏是 3.3V 器件）；④ 模块没带上拉电阻、接触不良。

解法

先 i2c.scan() 看实际地址；确认 SDA→GPIO21、SCL→GPIO22；用 3.3V 供电（别接 5V）；检查杜邦线接触；多数模块自带 4.7k 上拉，无需另加。

为什么两根线能连很多设备？因为每个 I2C 设备有唯一地址。ESP32 喊地址，匹配的设备才回应，其余保持沉默。这也是它比「一个设备占一堆脚」的 SPI 更省引脚的原因。

传感器实战：温湿度「看板」

时长 ≈ 2h · 中级阶段
🌫️
心智图像：传感器就是把物理世界「翻译」成电信号的小翻译官。BME280 这类「环境传感器」内部量好温度/湿度/气压，然后通过 I2C 把数字报到 ESP32。你只管问它「现在多少度」，它答你一个数字。

把前面学的 I2C（课 5）+ ADC（课 1）组合起来，就能做环境监测。下面用 BME280（温/湿/压）为例，读出来再画到 OLED 上——一个会自己显示天气的小摆件。

from machine import I2C, Pin
from bme280 import BME280
from ssd1306 import SSD1306_I2C
import time

i2c = I2C(0, scl=Pin(22), sda=Pin(21))
bme = BME280(i2c=i2c)
oled = SSD1306_I2C(128, 64, i2c)

while True:
    t, p, h = bme.read_compensated_data()
    oled.fill(0)
    oled.text("Temp %.1f C" % t, 0, 0)
    oled.text("Humi %.1f %%" % h, 0, 16)
    oled.text("Pres %.0f hPa" % (p/100), 0, 32)
    oled.show()
    time.sleep(2)
🛠️排错卡 · 传感器读不到 / NaN
症状

读出来是 NaN、一直是 0，或导入库直接报错。

原因

① 驱动库（ssd1306/bme280/dht）没上传到板子；② I2C 地址冲突；③ 接线松 / 供电不足；④ DHT 时序严格，偶尔失败。

解法

把驱动库放进板子 /lib 目录；先 i2c.scan() 确认地址；新手直接用 BME280（I2C，稳）；检查 3.3V 供电与接触。

DHT 还是 BME？DHT11/22 用一根线（单总线），便宜但慢、易失败；BME280/BME680 用 I2C，更稳更准，还能测气压。新手建议直接上 BME280，体验好很多。

定时器与中断：不用一直盯着，事件来了再响应

时长 ≈ 2.5h · 中级阶段
⏰
心智图像：前面 time.sleep() 像你闭眼数秒、啥也不干地干等。中断和定时器则是「定个闹钟 / 雇个门铃」——时间到了或按钮被按，芯片自动跳去执行一段「回调函数」，平时该干嘛干嘛。这叫非阻塞，效率天差地别。

两种常用手段：定时器（Timer）周期性触发；引脚中断（Pin.irq）在电平变化时触发（比如按钮按下瞬间）。两者都让你摆脱「死等」。

from machine import Timer, Pin

led = Pin(2, Pin.OUT)
tim = Timer(0)

# 每 500ms 翻转一次灯(非阻塞呼吸式闪烁)
def toggle(t):
    led.value(1 - led.value())

tim.init(period=500, mode=Timer.PERIODIC, callback=toggle)

# 按钮中断: 按下瞬间才触发, 平时主循环空转
btn = Pin(4, Pin.IN, Pin.PULL_UP)
def on_press(p):
    print("按钮被按下了!(在中断里)")
btn.irq(trigger=Pin.IRQ_FALLING, handler=on_press)

while True:
    pass   # 主循环可以去做别的事
中断里别做「重活」回调函数应当短平快（设个标志位、翻转个变量）。在里面用 time.sleep 或打印大量内容都可能导致系统不稳定。正确姿势：中断里只「记一笔」，主循环里再处理。
🛠️排错卡 · 中断不触发 / 卡死
症状

按钮按了中断没反应，或进中断后整个程序卡住。

原因

① 没设 handler 或 trigger 方向写反；② 回调函数里调了阻塞函数（time.sleep、大量 print）；③ 引脚不支持中断。

解法

用 btn.irq(trigger=Pin.IRQ_FALLING, handler=on_press)；回调里只设标志位，主循环再处理；确认所用 GPIO 支持中断（几乎都支持）。

深度睡眠与省电：电池供电的关键

时长 ≈ 1.5h · 中级阶段
🔋
心智图像：ESP32 有三种「清醒程度」。全速工作像跑步（费电）； modem‑sleep 像站着打盹（Wi‑Fi 还连着，偶尔睁眼）；深度睡眠像冬眠——只留一个心跳( RTC )维持，电流从几十 mA 掉到几微安。电池设备靠它续命几个月。

深度睡眠时 CPU、Wi‑Fi、大部分外设都断电，只有 RTC 和少量内存活着。可以设定「被某个引脚唤醒」或「定时唤醒」。

import machine, time
# 唤醒后先打印一次, 然后睡 10 秒再醒
print("醒啦! 原因:", machine.wake_reason())

# 方式1: 定时唤醒(毫秒)
machine.deepsleep(10000)

# 方式2: 引脚唤醒(需外部电路把 GPIO 拉低) 
# machine.pinwake = Pin(4, mode=Pin.IN)
# machine.deepsleep()
交互 · 深度睡眠省电对比拖滑块看续航变化
🛠️排错卡 · 深度睡眠唤不醒
症状

进了深度睡眠就再也醒不过来，或醒来后立刻又睡。

原因

① 唤醒源没接对（需外部把 GPIO 拉低）；② ext0/ext1 配置错；③ 唤醒后没根据 wake_reason() 做该做的事就又睡了；④ 没共地。

解法

用 machine.wake_reason() 确认唤醒源；按键一端接 GPIO、一端接 GND，配置 ext0(pin, level=0)；确保与 ESP32 共地；醒来先做采集/上报再睡。

深度睡眠的「代价」睡死状态下你的程序变量全没了、网络也断了，醒来相当于「重新启动」。所以省电设备要设计成：醒来 → 快速采集/上报 → 立刻再睡。这正是物联网传感器的经典节奏。
