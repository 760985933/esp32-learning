MicroPython 异步：用 uasyncio 同时做多件事

时长 ≈ 2h · 高级阶段
🌀
心智图像：真正的设备要一边维持网页、一边读传感器、一边闪灯。如果还用 while True 硬等，一件事卡住全完蛋。uasyncio 就是 MicroPython 里的「并发管家」——你用 async/await 写几个「可暂停的任务」，它负责在它们之间聪明地来回切换。好消息：这和你在 Python 里学的 asyncio 几乎一模一样！

你会的东西直接迁移：async def、await、asyncio.create_task()、asyncio.sleep()（注意它是非阻塞的，不是 time.sleep）。

import asyncio, machine
led = machine.Pin(2, machine.Pin.OUT)

async def blink():                # 任务1: 闪灯
    while True:
        led.value(1); await asyncio.sleep(0.5)
        led.value(0); await asyncio.sleep(0.5)

async def read_sensor():          # 任务2: 读传感器(互不阻塞)
    while True:
        print("读数..."); await asyncio.sleep(2)

async def main():
    await asyncio.gather(blink(), read_sensor())

asyncio.run(main())
和「中断 + 定时器」的区别？定时器/中断是「硬件触发的小纸条」，适合极简响应；uasyncio 是「软件层的协作式多任务」，适合把一整套逻辑（网络、传感器、UI）组织得清晰可读。复杂项目两者常配合使用。
🛠️排错卡 · uasyncio 卡死 / 只跑一个任务
症状

程序卡住不动，或只执行了其中一个任务。

原因

① 协程里忘了写 await；② 用了阻塞的 time.sleep() 而不是 asyncio.sleep()；③ 没用 asyncio.gather 并发，主任务没 asyncio.run。

解法

协程内一律用 await asyncio.sleep()；所有耗时调用前加 await；用 asyncio.gather(a(), b()) 并发；入口用 asyncio.run(main())。

ESP‑NOW：芯片之间直接「喊话」，无需路由器

时长 ≈ 2h · 高级阶段
📡
心智图像：Wi‑Fi 通常要「先连路由器才能说话」。ESP‑NOW 是 ESP32 的暗语频道：两块板子可以直接互发数据包，不经过路由器。延迟极低、省电，特别适合「一个主控 + 一圈传感器节点」的组网。

下面是一块「发送端」给对端广播消息的骨架。双方都要 import espnow，接收端把自己的 MAC 地址（或广播地址）加为 peer。

import network, espnow
w = network.WLAN(network.STA_IF); w.active(True)
e = espnow.ESPNow(); e.active(True)

# 把「接收端」的 MAC 加为对端(广播可写 b'\xff'*6)
peer = b'\xaa\xbb\xcc\xdd\xee\xff'
e.add_peer(peer)

while True:
    e.send(peer, "温度:23.5")   # 直接喊话
    time.sleep(1)

# 接收端: msg = e.recv()  → (mac, data)
交互 · ESP‑NOW 一对多广播拖滑块改节点数
ESP‑NOW vs Wi‑Fi STA两者可共存于同一块板（都走 2.4GHz 射频），但 ESP‑NOW 包更简单、没有 TCP/IP 那套握手，所以快且省电。需要「跨网络/上云」时还是得用普通 Wi‑Fi。
🛠️排错卡 · ESP‑NOW 收不到
症状

发送端发了，接收端 recv() 一直等不到。

原因

① 双方 MAC 写错 / 广播地址不对；② 信道不一致；③ 收发端都要 import espnow 并 e.active(True)；④ 发送端没 add_peer。

解法

确认双方 STA 都 active(True)；peer 用正确 MAC 或 b'\xff'*6 广播；保证同一 Wi‑Fi 信道；收发端都 e.active(True) 且发送端已 add_peer。

MQTT：物联网的「消息总线」

时长 ≈ 2.5h · 高级阶段
🚌
心智图像：MQTT 像公司里的公告板（Broker 中介）。设备 A 不认识设备 B，但它往「主题 home/living/temp」上贴（publish）一条消息；任何订阅（subscribe）了这个主题的设备都会收到。解耦、跨网络、手机/云端都能参与——这是物联网最主流的协议。

ESP32 连上 Wi‑Fi 后，用 umqtt.simple 连接公共或私有的 Broker（如 EMQX、Mosquitto、或免费测试 broker）。下面把温湿度发到主题，手机端随便什么 MQTT 客户端订阅即可看到。

import ubinascii, ujson
from umqtt.simple import MQTTClient

c = MQTTClient("esp32_01", "broker.emqx.io", 1883)
c.connect()

while True:
    msg = ujson.dumps({"t": 23.5, "h": 55})
    c.publish("home/living/temp", msg)   # 贴公告
    # c.subscribe("home/cmd") 可订阅手机下发的指令
    time.sleep(5)
主题（Topic）的命名套路用斜杠分层，像文件夹：家/房间/设备/属性。这样你既能订阅「某一盏灯」，也能订阅「整个客厅/*」批量接收。善用通配符 #(多层) 和 +(单层)。
🛠️排错卡 · MQTT 连不上 / 收不到
症状

connect() 失败，或发布后订阅端收不到。

原因

① broker 地址/端口错（默认 1883）；② 板子还没连上 Wi‑Fi；③ 订阅端没先 subscribe 就等；④ 主题大小写/拼写敏感不一致。

解法

先用公共测试 broker（broker.emqx.io:1883）；确认已 wlan.isconnected()；发布前先 connect()；订阅端先 subscribe 再循环收；主题两端逐字一致。

BLE 蓝牙低功耗：直连你的手机 App

时长 ≈ 2.5h · 高级阶段
📲
心智图像：BLE 像 ESP32 开了一个「会客室」（GATT 服务），里面摆了几张「桌子」（特征 Characteristic）。手机走近，连上门，就能「读/写」桌子上的数据——比如读电量、写一句「开门」。和经典蓝牙比，它大部分时间睡觉、只在通信瞬间醒一下，所以极省电。

MicroPython 用 bluetooth + ble 模块，按 GATT 规范注册「服务/特征」。下面把一颗「读数」特征暴露出去，手机用任意 BLE 调试 App 即可读取。

import bluetooth, machine
from ble import bluetooth  # 实际为: from micropython import const ... (示意)
ble = bluetooth.BLE(); ble.active(True)

# 简化示意: 注册一个可读特征, UUID 自定义
SERVICE = bluetooth.UUID(0x1234)
CHAR    = bluetooth.UUID(0x5678)
CHAR = (CHAR, bluetooth.FLAG_READ | bluetooth.FLAG_NOTIFY)
SERVER = (SERVICE, [CHAR])
handles = ble.gatts_register_services([SERVER])

# 手机读时, 回一个当前读数
ble.gatts_write(handles[0][0], b"23.5")
BLE 还是经典蓝牙？ESP32 两种都支持，但做手机近场控制/手环类，用 BLE（省电、iOS/Android 原生支持）；传音频/大文件才用经典蓝牙。新手别一上来就折腾音频。
🛠️排错卡 · BLE 手机搜不到 / 无数据
症状

手机用 nRF Connect 搜不到设备，或连上后读不到数据。

原因

① GATT 服务没注册成功；② UUID 与手机端不一致/冲突；③ 手机用的是经典蓝牙 App（BLE 需专用）；④ iOS 有连接缓存。

解法

确认 ble.active(True) 且 gatts_register_services 成功；用标准 128 位 UUID；手机用 BLE 调试 App（nRF Connect）；iOS 上删掉设备重连清缓存。

电机控制：让设备「动起来」

时长 ≈ 2.5h · 高级阶段
⚙️
心智图像：GPIO 引脚像手指，力气很小（几十 mA），推不动电机这种「大块头」。于是我们要雇一个壮汉（驱动板 / MOSFET / 电机驱动芯片）：手指只给壮汉发「轻推/重推」的信号（PWM 调占空比 = 控制转速），壮汉才去驱动电机。舵机则更简单——给它一个 PWM 脉冲宽度，它就转到对应角度。

直流电机

用 L298N / TB6612 / MOSFET 驱动。PWM 调转速；换两根线顺序调方向。

舵机 Servo

PWM 脉宽 0.5~2.5ms 对应 0~180°。MicroPython 有 machine.Servo。

步进电机

一步一步走，定位精准。需驱动板（A4988 等）给脉冲序列。

from machine import PWM, Pin
import time
motor = PWM(Pin(14), freq=1000)  # 接驱动板 PWM 输入

motor.duty_u16(32768)   # 约 50% → 半速
# 舵机:  from machine import Servo; s=Servo(Pin(13)); s.angle(90)
🛠️排错卡 · 电机不转 / 只响 / 单向
症状

电机完全不转，或只嗡嗡响不转，或只能一个方向。

原因

① 驱动板没接独立电源或与 ESP32 没共地；② IN1/IN2 逻辑错（同高同低都不转）；③ PWM 频率/占空比不对；④ 电机电流超过驱动板能力；⑤ 缺续流二极管烧了驱动芯片。

解法

驱动板接独立电源并和 ESP32 共 GND；IN1/IN2 一高一低控制方向、都低则刹车；PWM 调占空比调速；确认驱动板电流够用；务必加续流二极管。

电机的反向电压会「击穿」芯片电机是感性负载，断电瞬间会产生高压反冲。务必加续流二极管，并用独立电源/驱动板，绝不用 GPIO 直接驱电机。否则 ESP32 很可能当场去世。

其它传感器：红外 / 触摸 / 霍尔

时长 ≈ 1.5h · 高级阶段
🧲
心智图像：传感器就是各种「感官」：触摸像皮肤（ESP32 好多脚自带触摸感应，手一碰就变）；霍尔像能感知磁场的舌头（靠近磁铁就触发，用来测转速/门磁）；红外像眼睛（发射管+接收管，有无遮挡=对射/反射式测距、循迹）。
  • 触摸 Touch：machine.TouchPad(Pin(32))，read() 返回电容值，手靠近数值骤降。无需外接元件！
  • 霍尔 Hall：ESP32 内核集成霍尔传感器，import machine; machine.hall_sensor() 读磁场。外接开关型霍尔（如 3144）更常用，输出就是个数字引脚。
  • 红外 IR：用红外接收头（VS1838）+ ir 库解码遥控器；或一对红外对管做循迹/避障。
from machine import TouchPad, Pin
tp = TouchPad(Pin(32))
while True:
    v = tp.read()
    if v < 200:     # 数值低于阈值 = 被触摸
        print("摸到啦!")
    time.sleep(0.1)
为什么唤醒用触摸/霍尔很常见？它们功耗极低、响应快，常配合「深度睡眠」做「轻触即醒」的电池设备（回想中级课 8 的省电模型）。
🛠️排错卡 · 触摸/霍尔/红外 不起作用
症状

触摸读数一直不变；霍尔接了没反应；红外遥控收不到。

原因

① 触摸脚只能用带 TouchPad 功能的脚（不是所有脚都行）；② 霍尔模拟型需 ADC 脚、开关型是数字脚，接错类型读不到；③ 红外接收头 OUT 接错脚、或被强光干扰；④ 阈值设得不合理。

解法

触摸用 GPIO32/33/27 等专用脚；霍尔按「模拟/数字」选对引脚类型；红外 OUT 接任意数字 GPIO、远离日光灯；先串口打印原始读数再定阈值。

OTA 无线升级：不用插线就能更新固件

时长 ≈ 2h · 高级阶段
🛰️
心智图像：设备装到天花板上之后，你还想爬梯子插 USB 去改代码吗？OTA（Over‑The‑Air）就是「空中投递」：设备连网后，从服务器下载新版本程序，写进 Flash，重启即生效。像手机系统更新一样省事。

MicroPython 下 OTA 思路通常是：把代码放在 /lib、用「版本文件 + 下载替换」策略，或上 ESP‑IDF 后使用官方双分区（A/B）OTA。下面给一个最简「下载新 main.py 并重启」的骨架：

import urequests, machine, uos
# 1) 从你的服务器拉取新版本
r = urequests.get("http://myserver/main.py")
# 2) 写回文件系统(可先备份旧版)
uos.rename("main.py", "main.bak")
with open("main.py", "w") as f: f.write(r.text)
# 3) 重启生效
machine.reset()
生产级 OTA 的关键三件事① 校验下载完整性（哈希/签名），防止半截损坏变砖；② 双备份，新版本起不来还能回滚旧版；③ 分批灰度，别一次推给所有设备。这些在专家阶段「量产工程化」会展开。
🛠️排错卡 · OTA 升级失败 / 变砖
症状

下载后设备起不来、反复重启，或直接「变砖」连不上。

原因

① 下载只写了一半就重启（网络中断）；② 新代码有语法/导入错误；③ MicroPython 单文件覆盖没有备份，坏了无法回退；④ 没校验完整性。

解法

升级前先 uos.rename 备份旧版；下载完校验长度/哈希再写；写失败保留旧版；上 ESP‑IDF 后用官方 A/B 双分区，起不来自动回滚。
